## Часть I. Что мы вообще считаем (постановка для новичка)

### 1.1 Физика: коагуляция Смолуховского

Представьте «суп» из частиц разных размеров (капли, агрегаты, аэрозоли). Частицы
случайно сталкиваются и **слипаются** (коагулируют): частица размера `i` + частица
размера `j` → частица размера `i+j`. Со временем мелких частиц становится меньше, а
крупных — больше; распределение размеров эволюционирует.

Скорость слипания пары `(i,j)` задаётся **ядром коагуляции** `K(i,j)` — функцией, которую
диктует физика процесса (броуновское движение, гравитация, турбулентность...). В нашем
бенчмарке используется «атмосферное» ядро `ker_atmos(i,j)` (формула в §3.10).

Классически это описывается уравнением Смолуховского (система ОДУ на концентрации
`c_k = N_k/V`, где `N_k` — число частиц размера `k`, `V` — объём). Но при многих размерах
прямое решение ОДУ дорого и плохо ловит дискретную/стохастическую природу. Поэтому —
**Монте-Карло**.

### 1.2 Метод: стохастическая симуляция (kinetic Monte Carlo)

Мы моделируем сам случайный процесс «по событиям». Состояние — вектор счётчиков
`N = (N_1, N_2, ...)`. Истинная скорость акта `(i,j)→i+j`:

```
R(i,j) = K(i,j) · N_i · N_j / (2V)
```

Точный алгоритм (Gillespie / SSA): на каждом шаге (а) разыграть время до следующего
события `dt ~ Exp(Λ)`, где `Λ = Σ R(i,j)` — суммарная скорость; (б) выбрать пару `(i,j)`
с вероятностью `R(i,j)/Λ`; (в) применить её (`N_i--, N_j--, N_{i+j}++`); (г) повторять,
пока время не дойдёт до `t_max`.

Проблема: пар `(i,j)` — порядка `N²` (миллионы размеров → триллионы пар). Считать `Λ` и
выбирать из триллионов вариантов на каждом шаге невозможно. Нужно умное разложение.

### 1.3 Mosaic-Skeleton: блоки и мажоранты

Матрицу скоростей `R` размера `N×N` разбивают на **блоки** (Mosaic-Skeleton разложение —
отсюда название библиотеки MSk). Диагональные блоки маленькие и плотные; далёкие от
диагонали — крупные, но «гладкие» (низкоранговые). Это даёт компактное представление.

Ключевой приём — **мажоранта** (верхняя оценка). Для каждого блока `b` берём
`rawmax_b = max K(i,j)` по блоку. Тогда **интенсивность блока**

```
I_b = rawmax_b · (Σ N_i по строкам блока) · (Σ N_j по столбцам блока)
    = rawmax_b · row_sum_b · col_sum_b
```

является верхней оценкой суммарной скорости внутри блока (`I_b ≥ Σ_{(i,j)∈b} R(i,j)·2V`).
Сумма по блокам `Σ I_b` мажорирует истинную `Λ·2V`.

### 1.4 Accept-Reject per block (сердце алгоритма)

Вместо выбора из триллионов пар напрямую — двухуровневая выборка с отказами:

- **A.** Разыграть время: `λ = (Σ I_b)/(2V)`, `dt ~ Exp(λ)`, `t += dt`.
- **C1.** Выбрать блок `b` пропорционально его интенсивности `I_b`.
- **C2.** Внутри блока выбрать строку `i` пропорционально `N_i` и столбец `j`
  пропорционально `N_j`.
- **C3.** Принять пару с вероятностью `K(i,j)/rawmax_b` (≤1, т.к. `rawmax` — максимум по
  блоку). Если отказ — событие «пустое» (reject), состояние не меняется.

Почему это даёт **точный** результат? Эффективная скорость принятого `(i,j)`:

```
λ · P(блок b) · P(i|b) · P(j|b) · P(accept)
 = [ΣI/2V] · [I_b/ΣI] · [N_i/row_sum_b] · [N_j/col_sum_b] · [K/rawmax_b]
```

Подставляя `I_b = rawmax_b·row_sum_b·col_sum_b`, всё лишнее **сокращается** до
`K(i,j)·N_i·N_j/(2V) = R(i,j)`. То есть мажоранта влияет только на долю отказов
(эффективность), но не на распределение принятых событий (корректность). Этот принцип
— классическое **прореживание Пуассона** (Lewis–Shedler / Ogata thinning) — окажется
центральным в самой мощной оптимизации (N7, §3.11).

### 1.5 Структуры данных

Чтобы C1/C2 были быстрыми, нужны структуры для «выбрать элемент пропорционально весу» с
быстрым обновлением весов:

- **Сегментное дерево сумм с ленивым масштабированием** (`SegTreeLazy`). Дерево, где лист
  хранит вес, а внутренний узел — сумму поддерева. Выбор пропорционально весу = спуск от
  корня (`O(log N)`); обновление листа = подъём к корню (`O(log N)`). «Ленивое
  масштабирование» (`lazy_scale`) позволяет умножить ВСЕ веса на константу за `O(1)`
  (нужно при rescale, §1.8).
  - `block_intensity_tree` — дерево интенсивностей блоков (для C1).
  - `n_store_tree` — дерево счётчиков `N_k` (для C2: выбор частицы внутри блока).
- **Кэши сумм** `row_sum_cache`, `col_sum_cache` в каждом блоке (`BlockInfo`) — чтобы не
  пересчитывать суммы по дереву каждый раз.
- **BlockIndex** — для данной строки/столбца быстро найти все блоки, её содержащие
  (нужно при обновлении, §1.6).

### 1.6 Что происходит на accept (фаза D2)

Когда пара принята (`i+j → k`), надо обновить состояние:

- **D2.1** — обновить `n_store_tree` в 3 листьях (`i, j, k`).
- **D2.2** (`apply_delta`) — для каждого блока, чью строку/столбец задело (`i, j, k`),
  обновить `row_sum_cache`/`col_sum_cache`. Таких «грязных» блоков ~80–160 за accept.
- **D2.3** — пересчитать интенсивности грязных блоков и **пропагировать** изменения вверх
  по `block_intensity_tree`, чтобы C1 следующего шага сэмплировал из актуального дерева.

Эти три под-фазы (особенно D2.3) и окажутся главными «пожирателями» времени.

### 1.7 Масштабирование популяции (rescaling)

Коагуляция уменьшает число частиц. Когда оно падает вдвое (`current_total ≤ N₀/2`),
статистика деградирует. Тогда систему «удваивают»: все счётчики `×2`, `V ×2`,
интенсивности `×4` (через `O(1)` ленивое масштабирование деревьев). Это стандартный
контроль популяции; счётчик называется `Scalings`. При `t_max=1e7` это случается ~18 раз.

### 1.8 Как меряем и что считаем успехом

- **Wall** — реальное время прогона (главная метрика). Шум на кластере ~1%; изменение
  <1% = шум.
- **Frobenius** — норма отклонения посчитанного распределения от эталонного решения
  (`||n_sim − n_ref|| / ||n_ref||`). Контроль корректности: при одном seed должна
  совпадать до ~5 знака.
- **Фазовый профиль** через `__rdtsc()` (счётчик тактов процессора) — доля и `cyc/call`
  по фазам A/C1/C2/C3/D1/D2.1/D2.2/D2.3. Накладные ~6–13% (одинаково для всех вариантов,
  так что сравнение корректно). Есть два бинаря: **prof** (с rdtsc + Frobenius) и **fast**
  (голая скорость).
- **Критерий «бит-в-бит»:** events, accepted и Frobenius идентичны baseline. Большинство
  оптимизаций удовлетворяют ему (одна — N7 — нет, см. §3.11).

## Часть II. Природа узкого места (почему это важно понять до оптимизации)

### 2.1 Железо

Целевая машина — HPC-кластер на Intel Xeon Gold 6226R (Cascade Lake), 32 ядра/нода,
384 ГБ RAM. Иерархия памяти и примерные латентности (в тактах CPU):
L1 (32 КБ/ядро) ~4–5; L2 (1 МБ/ядро) ~14; L3 (22 МБ/сокет) ~40–50; DRAM ~200+.
Доступ к памяти тем дороже, чем дальше уровень.

### 2.2 Главный диагноз: мы упёрты в ЛАТЕНТНОСТЬ ЗАВИСИМЫХ ЦЕПОЧЕК

Аппаратные счётчики (perf_event) на оптимизированной версии дали характерную картину:

- **IPC ≈ 1.75** (инструкций за такт) при теоретическом потолке ~4 и реальном для
  pointer-chasing ~2. Процессор простаивает.
- **stall_total ≈ 21–23%** тактов — ядро вообще ничего не выполняет (ждёт память).
- **L1D-miss-stall ~7–12%, L2-miss-stall ~4–8%** — простои в ожидании L2/L3.
- **LLC-miss ≈ 1.7%** (мало!) — рабочий набор помещается в L3. Значит **НЕ упираемся в
  пропускную способность DRAM**.

Вывод (мы обозначали его «H4»): узкое место — **латентность зависимых обращений к памяти**
при спусках по деревьям и обновлении блоков (pointer-chasing: чтобы сделать следующий шаг,
надо дождаться предыдущего чтения). Это НЕ нехватка вычислений, НЕ нехватка полосы памяти.

### 2.3 Что из этого следует (стратегия)

Это сразу отсекает целые классы «очевидных» идей и подсказывает работающие:

- **Работает:** укоротить цепочки зависимых чтений (более плоские/высокоарные деревья);
  держать несколько независимых промахов «в полёте» одновременно (memory-level
  parallelism, MLP); сократить число инструкций (codegen).
- **НЕ работает:** уменьшить тип данных / footprint (латентность ≠ объём); hugepages /
  NUMA / увеличение полосы (мы не bandwidth-bound); гнаться за IP→4 (невозможно для
  зависимых цепочек).

Несколько неудач ниже — это ровно нарушения этого принципа (например, `float`-деревья N2
или SoA opt-11). Понимание природы узкого места — половина успеха.

## Часть III. Хронология: что внедряли, что сработало, что нет

> Важная методологическая деталь про абсолютные числа. Часть ранних замеров делалась на
> «reference»-машине (legacy 140 с → opt-7 78 с), но потом выяснилось, что на HPC-кластере
> картина hot-path иная (см. opt-7 ниже — сюрприз ×20). Поэтому **канонические числа —
> кластерные**, `V=10⁷, t_max=10⁶`. Где числа из разных режимов, это помечено. Надёжны
> прежде всего **относительные Δ**.

### Эра 0 — чистое переписывание (legacy → opt-0): −34%

Исходный «profiled»-код был усеян отладочной обвязкой. `opt-0` — чистая версия
(`particle_sampling_system_ar.hpp`): быстрый ГПСЧ Xoshiro256++ вместо `std::mt19937`,
ручной `[0,1)` без `uniform_real_distribution`, ленивое `scaleAll` за `O(1)`,
дедупликация грязных блоков, шаблонное ядро (инлайнится) вместо `std::function`. Уже
**−34%** (на reference: 140→92 с). Урок: прежде чем «оптимизировать», уберите мусор.

### Эра 1 — дешёвые победы (компилятор и арифметика)

**📊 Что мотивировало:** на этом этапе профилирование было ещё грубым — счёт горячих
операций. Подсчитали: `std::log` (для `dt`) зовётся **~334M раз/прогон** → отсюда ffast-math
(opt-3); делений в `SegTreeLazy.update` **~7.8 млрд** → отсюда `inv_lazy_scale` (opt-4).
А главный сюрприз эры — opt-7 — пришёл уже от полноценного фазового профиля (см. ниже).

- **opt-1 (LTO) — REVERTED, шум.** Hot-path и так шаблонно-инлайнен, MSk-вызовы вне цикла.
- **opt-2 (убрать progress bar) — шум,** оставлен как UX (срабатывал ~16 раз за прогон).
- **opt-3 (`-ffast-math`) — ✅ −7.9% (ref) / −6% (cluster).** `std::log` (для `dt`)
  зовётся ~334M раз; fast-math включает быстрый путь libm. Events/Frobenius идентичны.
- **opt-4 (`inv_lazy_scale`) — ✅ −4.7% (ref) / −1.25% (cluster).** В `SegTreeLazy.update`
  деление на масштаб заменено умножением на предвычисленную обратную величину (~7.8 млрд
  делений → умножения). Обратное пересчитываем только при rescale (15 раз).
- **opt-6 (PGO, reference) — ✅ −3.3%.** Profile-guided optimization: компилятор
  специализирует горячие ветки по частотам. На reference дало 1.79× кумулятивно. (На
  кластере PGO позже провалится — см. opt-15.)
- **opt-7 (кэш `row_sum`/`col_sum`) — СЮРПРИЗ: −1% (ref), но ✅ −19.5% (cluster!).**
  Вместо чтения сумм из дерева — инкрементальный кэш в `BlockInfo`. На reference под PGO
  чтения дерева были уже почти бесплатны (1%), а на кластере **без** PGO они дорогие, и
  кэш дал ×20 больше. **Урок: профиль зависит от машины; меряйте на целевом железе.**
- **opt-8 (force-inline `sampleFromBlock`/`sampleIndex`) — ✅ −0.86%.** Компилятор при -O3
  не инлайнил их сам; `__attribute__((always_inline))` дал C2 −3.3%.
- **opt-9 (сдвиг вместо деления) — ✅ −3%.** Размеры блоков — степени двойки, поэтому
  `(base+start)/block_size` заменено на `>>` с предвычисленным `log2`. C2 −18.7%.

### Эра 2 — атака на главные hotspot-ы (и серия поучительных провалов)

**📊 Профиль, который запустил всю охоту (opt-7, через `__rdtsc()`, `t_max=1e6`):**

| Фаза | доля wall | cyc/call | вызовов |
|---|---:|---:|---:|
| **C: sampleParticlePair** | **51%** | 835 | 426M |
| — C1 (спуск по block-дереву) | 18% от C | 133 | 426M |
| — C2 (2 спуска по `n_store`) | 38% от C | 279 | 426M |
| — C3 (ядро + accept-reject) | 43% от C | 315 | 426M |
| **D2: accept + update** | **46%** | 4015 | 79M |
| — D2.1 (обновление `n_store`) | 4% от D2 | 152 | 79M |
| — D2.2 (`apply_delta`) | 27% от D2 | 985 | 79M |
| — **D2.3 (пропагация интенсивностей)** | **68% от D2 ≈ 28% всего CPU** | **2465** | 79M |
| D1 (отказ) / A (время) | 2.6% / 2.1% | — | — |

(Замечание: `gprof` тут врал — из-за инлайнинга атрибутировал 70% в `load_reference_solution`,
которая зовётся 1 раз ПОСЛЕ симуляции; `callgrind` давал ~600× замедление. Поэтому —
ручной `__rdtsc()` по фазам, ~6% накладных, одинаковых для всех.)

**Вывод из таблицы прямой:** D2.3 — крупнейший single hotspot (**28% всего CPU**), это
`for(bid : dirty) tree.update(...)` — ~16 уникальных блоков × 17 уровней дерева, memory-bound.
Второй — D2.2. Отсюда и серия атак: **по D2.3** (opt-5, opt-12, opt-14), **по D2.2** (opt-11,
opt-13a). Многие провалились (и это ценно):

- **opt-5 (батч D2.3 через `std::sort`) — ❌ +41%, REVERTED.** Теоретически `K log K + 2K`
  лучше `K log B`, но overhead `std::sort` (60–80 элементов, mispredict) и тот факт, что
  ~50% событий — отказы (без обновления), убили идею. Урок: алгоритмически «лучше» ≠
  быстрее на маленьких `K`.
- **opt-5' (bitset вместо `visited_gen`) — ❌ +6%, REVERTED.** Гипотеза «`visited_gen`
  промахивается в L2» неверна: горячие блоки и так в L1. Bitset добавил операций.
- **opt-10 (3× `cbrt` вместо 4× `pow` в ядре) — ❌ +0.6%, REVERTED.** `-ffast-math` уже
  сам заменяет `pow(x,1/3)→cbrt`. **Урок: смотрите ассемблер перед «очевидной»
  оптимизацией трансцендентных функций.**
- **opt-11 (SoA: вынести кэши сумм в отдельные массивы) — ❌ +3.3%, REVERTED.** Гипотеза
  «8× плотность кэша». Реальность: `apply_delta` стал читать 3–4 раздельных потока вместо
  одного; `BlockInfo` (AoS) и так влезал в 1 кэш-линию → SoA только умножил промахи.
  **Это прямое следствие §2.2: мы latency-bound, и дробление структуры вредит.**
- **opt-11b (перестановка полей в `BlockInfo`) — ❌ 0%, REVERTED.** HW-prefetcher Cascade
  Lake и так подтягивает соседнюю линию. Косметика, добавляющая когнитивную нагрузку.
- **opt-12 (батч ленивой пропагации D2.3, БЕЗ sort) — ✅ −4.82%.** Вместо `N` независимых
  `update` (каждое поднимается через `log` уровней) — сначала пишем все грязные листья,
  потом BFS вверх по уровням, трогая каждый внутренний узел **ровно один раз** (через
  `visited_internal == gen`). Близко к корню все грязные блоки делят предков — там главная
  экономия. D2.3 −18.4%. (Та же идея, что провалившийся opt-5, но без `std::sort`.)
- **opt-13a (плоский CSR для `BlockIndex`) — ✅ −0.77%.** Было: `vector<vector<int32_t>>`
  (N независимых аллокаций, ~216 МБ метаданных). Стало: после построения — плоский CSR
  (`row_data[]` + `row_offsets[]`), scratch выбрасывается. D2.2 −11.1%, память 216→16 МБ.
  Гипотеза «освобождение L3 даст большой косвенный выигрыш» НЕ подтвердилась (горячие
  данные и так в кэше) — выигрыш только от одной устранённой кэш-линии на доступ.
- **opt-13 (аналитическая формула block-id) — ❌ CRASH, REJECTED.** Попытка убрать
  `BlockIndex` арифметикой. Структурное предположение о раскладке блоков оказалось неверным
  на краевых случаях → «double free». **Урок: проверяйте партиционные допущения отдельным
  sanity-тестом до правки hot-path.**
- **opt-13b (`min_block = 8` по умолчанию) — ✅ −2.6%.** Свип по min_block (минимальный
  размер блока): меньше блок → плотнее мажоранта → выше accept → меньше событий; но больше
  блоков → дороже D2. Оптимум — **mb=8** (216.7 с). Бесплатно сменой дефолта.
- **opt-14 (SW-prefetch в D2.3) — ❌ +7.2%, REVERTED.** Гипотеза «D2.3 memory-bound, надо
  предзагружать». Реальность: 3 prefetch × ~80 грязных + BFS = ~140 КБ загрязнения кэша на
  accept, L1 (32 КБ) полностью вытесняется. D2.3 +20.7%, ещё и соседи пострадали. **HW
  prefetcher уже справлялся.** Урок: ручной prefetch на хорошем HW-prefetcher часто вреден.
- **opt-15 (PGO на кластере) — ❌ ~0%, оставлен выключенным.** Меньшие фазы улучшились, но
  D2.3 регрессировал +7.7% (GCC переупорядочил горячие BFS-циклы, сломав L1i-локальность),
  всё скомпенсировав. На reference PGO давал −3.3%, на кластере — ноль.

Кумулятивно к концу Эры 2: opt-7 → opt-13b на кластере −11.9% (245.86 → 216.7 с).

### Эра 3 — перелом: «мы latency-bound» и структурные победы

Здесь мы аппаратными счётчиками установили §2.2 (latency-bound на зависимых цепочках) и
оптимизации стали бить точно в природу узкого места.

**📊 Что показали счётчики `perf_event` (на горячем пути):**

| метрика | значение | что значит |
|---|---:|---|
| IPC | **1.75** | при потолке ~4 — процессор простаивает |
| stall_total | **21–23%** тактов | ядро вообще ничего не выполняет (ждёт память) |
| L1D-miss-stall | ~7–12% | простой в ожидании L2/L3 |
| **LLC-miss** | **1.7%** | мало! рабочий набор в L3 ⇒ **НЕ упёрты в полосу DRAM** |

А line-level профиль (perfsample) показал, что **крупнейший кластер промахов — это
зависимые загрузки `tree[l]` в спуске C2** (и подъём D2.3). Диагноз: латентность зависимых
цепочек чтений. **Отсюда три направления, бьющие именно в это:** меньше инструкций на
цепочку (`-march`, opt-16); перекрыть независимые промахи (MLP, opt-17); укоротить саму
цепочку (B-арные деревья, opt-18/19). И — что сэкономило недели — отсюда же понимание,
ПОЧЕМУ float-дерево (N2) и SoA (opt-11) обязаны провалиться (они про объём, а не про
латентность).

- **opt-16 (`-march=native`) — ✅ −8.3%.** Базовый бинарь компилировался под generic
  x86-64 (0 AVX, legacy SSE-скаляр). `-march=native` под Cascade Lake включил AVX-512/FMA,
  `ctz` через `tzcnt` и т.п. C1 −30%, D2.1 −25%, D2.3 −12% (это codegen-фазы); C2/C3/D2.2
  почти не сдвинулись (они memory-latency, codegen там не помогает). **Frobenius идентичен
  — бит-в-бит.** Безопасно, т.к. машина сборки = машина запуска.
- **opt-17 (N1: пайплайнинг двух спусков C2) — ✅ −5.24%.** Спуски по строке `i` и столбцу
  `j` в C2 **независимы**. Новый `sampleTwoFromBlockShift` ведёт оба спуска в lockstep,
  держа **2 независимых промаха памяти в полёте** (MLP — пока ждём загрузку для `i`,
  процессор уже грузит для `j`). Порядок ГПСЧ сохранён → **бит-в-бит**. Это прямое
  применение §2.3: не убрать промахи, а перекрыть их. (Не путать с prefetch opt-14 — здесь
  реструктуризация зависимостей, а не спекулятивная подгрузка.)
- **N2 (float-дерево `n_store`) — НЕ внедрено, поучительный ноль.** Заменили `double` на
  `float` (16→8 МБ). Корректность ок, но wall **не сдвинулся**. Это эмпирическое
  доказательство §2.2: bottleneck — латентность зависимых цепочек, а не footprint/полоса.
  Половина типа не сокращает число зависимых обращений на спуск → 0 выигрыша. Правильный
  рычаг — **укоротить цепочку** (следующий пункт).
- **opt-18 (N3: 8-арное дерево интенсивностей) — ✅ −12.5%.** Главная идея Эры 3: бинарное
  дерево глубины `log₂N ≈ 20` заменено B-арным (B=8) глубины `log₈N ≈ 7`. Спуск (C1) и
  подъём (D2.3) проходят втрое короче цепочку зависимых L1→L2/L3-промахов. Создан отдельный
  класс `SegTreeBaryLazy` (старый `SegTreeLazy` оставлен для `n_store`). **Бит-в-бит** (с
  суммированием детей в том же порядке, что бинарное). В ходе верификации (вопрос «а
  совпадёт ли B=2 с бинарным?») нашли **2 бага**: (1) `expand_double` удваивал размер ДО
  выделения памяти → выход за границы при resize (недетерминизм!); (2) спуск вычитал
  лишнего ребёнка при fp-клампе. Оба исправлены. **Урок: «деградация к тривиальному
  параметру должна точно совпасть с эталоном» — мощный тест на корректность.**
- **opt-19 (арность B=8→B=4 + Option D) — ✅ −19% vs N1 (~146 с).** (а) Свип арности
  показал U-образную кривую с минимумом на **B=4**, а не B=8 (моя гипотеза «B=8 лучше, т.к.
  влезает в L3» НЕ подтвердилась: доминирует суммарная работа спуска ∝ глубина×арность +
  обработка узла, минимальна при малой арности). (б) **Option D:** объект `MosaicSkeleton`
  (и его фоновый рабочий поток из ThreadPool) живёт только на время построения геометрии и
  уничтожается ДО горячего цикла. Замер: системное время 46→4.5 с, в цикле строго **1
  поток**. Wall не изменился (поток был вне критического пути), но это критично для будущей
  цели «1 симуляция на ядро» (никаких лишних потоков).
- **N4 (B-арное дерево `n_store`) — ❌ +5%, REJECTED.** Та же идея, что opt-18, но для
  дерева частиц. Бит-в-бит (sanity B=2 ≡ baseline), но **медленнее**. **Урок —
  структурный:** B-арность помогает спускам ПОЛНОЙ глубины (дерево интенсивностей: всегда
  корень→лист ~20 уровней). Но спуск по `n_store` идёт на глубину `log(размер блока)`, а в
  мономер-тяжёлой симуляции блоки обычно мелкие (span=8) → для них B=4 читает 4 ребёнка
  ради выбора 1 + накладные на под-диапазон, что перевешивает экономию глубины. **Разным
  деревьям — разные структуры.**

### Эра 4 — мемоизация ядра (opt-20 / N6)

**📊 Что мотивировало:** после march/MLP/B-арности картина сместилась — переснятый профиль
opt-19 показал, что **C3 (ядро+AR) ≈ 18% wall** и в line-level всплыла строка `??:0` (libm)
на **6.6%** — это `std::pow`. Заглянули в код ядра: `ker_atmos(i,j)` делает **6 вызовов
`std::pow` на каждый вызов** × 312 млн вызовов.

Из них 4 — это `pow` от ЦЕЛОГО размера частицы (`i^{1/3}, j^{1/3}, i^{2/3}, j^{2/3}`). Их
закэшировали в таблицы
(`cbrt1[s]=pow(s,1/3)`, `pow23[s]=pow(s,2/3)`), которые растут вместе с системой.

- **opt-20 (N6: мемоизация ядра) — ✅ −6.6%, бит-в-бит.** 6 pow → 2 pow на вызов. C3
  281→222 cyc/call (−21%). Таблицы хранят те же значения `std::pow` → events/Frobenius не
  меняются (при условии использования и в C3, и в оценке `rawmax`). Ядро живёт в
  `examples/particles/`, библиотеку MSk не трогали. **Урок: я раньше оценивал

### Эра 5 — алгоритмический скачок: ленивая мажоранта (opt-21 / N7), −25%

Это самый крупный одиночный выигрыш и самая глубокая идея.

**📊 Что мотивировало:** несмотря на то, что N3 (B-арность) уже удешевил D2.3, переснятый
профиль opt-19/20 показал, что **D2.3 — СНОВА крупнейшая фаза (~25% wall, 43 с)**. Line-level
указал на точную строку: загрузка детей в `sum_children` при подъёме по дереву = **22% всех
сэмплов**, плюс запись грязных листьев = ещё 10%. То есть мы упёрлись в саму операцию
«пересчитать дерево интенсивностей на каждый accept». Дальше укорачивать дерево некуда
(B=4 — оптимум). Значит надо **не пересчитывать его каждый раз** — отсюда и родилась идея
ленивой мажоранты.

**Идея.** Дерево интенсивностей не обязано быть точным — оно используется для выборки
блока, а accept-reject (C3) и так корректирует. Значит можно хранить в дереве
**мажоранту** `Î_b ≥ I_b`, обновляемую **лениво**, а возникающее смещение компенсировать
**вторым accept-reject на уровне блока** (тем же приёмом, что `K/rawmax` в C3):

- В C1: выбрали блок ∝ `Î_b` → считаем точную `I_b` (это `O(1)`, кэши свежие) → принимаем
  блок с вероятностью `I_b/Î_b`; иначе ранний отказ (дёшево, до C2/C3).
- В D2.3: лист освежаем (и пропагируем) ТОЛЬКО если мажоранта пробита (`I_new > Î_b`) или
  слишком раздута (`Î_b > I_new·(1+m)²`) — двусторонняя полоса с гистерезисом. В типичном
  accept грязные блоки внутри полосы → **дерево не трогаем вообще**.

**Эффект:** дорогая пропагация D2.3 случается изредка вместо каждого accept. Измерено:
число обновлений листьев упало **в ~3650 раз** (966M → 264K на малом прогоне), фаза D2.3
**43.3 → 10.2 с (−76%)**. Цена: +7–14% «пустых» событий (ранние отказы в C1, ~+1–2% wall).
Нетто **−25%** (wall 142 → 107 с при margin=0.05).

**Доказательство несмещённости (это важно).** Через теорему прореживания Пуассона (Ogata):
тикаем со ставкой-мажорантой `Λ̂ = ΣÎ_b/(2V) ≥ Λ_true` и исполняем `(i,j)` с вероятностью
`R(i,j)/Λ̂` ⇒ исполненный поток в точности равен искомому марковскому процессу. Проверка:

```
P(исполнить (i,j) | тик)
 = [Î_b/ΣÎ]·[I_b/Î_b]·[N_i/row_b]·[N_j/col_b]·[K/rawmax_b] = K·N_i·N_j/ΣÎ = R(i,j)/Λ̂
```

`Î_b` **сокращается** ⇒ грубость мажоранты влияет лишь на КПД, не на закон. История
(`Î_b` зависит от прошлого) законна — прореживание допускает любую предсказуемую мажоранту.
Это двухуровневое прореживание: блочная мажоранта (чинит C1) вложена в ядерную (чинит C3).
Единственное новое условие — инвариант `Î_b ≥ I_b` всегда — доказывается индукцией и
сводится к **полноте списка грязных блоков**, на которой уже стоит точный алгоритм.

**Важная оговорка.** N7 добавляет лишний разыгрыш ГПСЧ ⇒ поток событий другой ⇒ **НЕ
бит-в-бит**. Поэтому корректность проверялась **статистически**: 50 сидов на `t_max=1e6`,
среднее Frobenius exact 0.151515 vs lazy 0.151553 — разница **+0.025% = 0.57σ**, дисперсия
не выросла ⇒ статистически неотличимо ⇒ несмещён. Это отход от прежнего строгого критерия,
поэтому N7 сделан **опциональным**: env `LAZY_MARGIN` (`<0` или не задан ⇒ выключено =
точный opt-20, бит-в-бит; `0.05` ⇒ −25%). Дефолтный бинарь остаётся проверяемым.

### Эра 6 — исчерпание одного потока (N8 — обе формы отклонены)

**📊 Что показал ре-профайл opt-21 (lazy) и куда он направил:**

| Фаза | opt-20 (exact) | opt-21 (lazy) | сдвиг |
|---|---:|---:|---|
| D2.3 (пропагация) | 43.3 с (25%) | **10.2 с (8%)** | **−76%** ← N7 сработал |
| D2.2 (`apply_delta`) | 29.4 с | 28.8 с (**22.7%, #1**) | ~= |
| C3 (ядро+AR) | 24.1 с | 24.2 с (19%) | ~= |
| C2 (спуск `n_store`) | 18.2 с | 16.7 с (13%/29% line) | ~= |

HW после N7: stall **22.7%** (стало ЕЩЁ более memory-bound — ушла compute-тяжёлая D2.3,
остался рассеянный доступ D2.2), LLC по-прежнему 1.7%. **Новый лидер — D2.2** (рассеянное
чтение блоков), затем C3 (ядро), C2 (зависимый спуск). Все три уже атакованы ранее
(opt-11/13a по D2.2, N6 по ядру, N4 по C2) и упёрты. C-фаза = 55% wall и масштабируется с
ЧИСЛОМ событий ⇒ единственный неиспробованный рычаг — **снизить число событий** (выше
accept). Это и проверил N8 (обе формы):

- **N8a (сделать `rawmax` тоньше) — ❌ невозможно.** `BoundaryScan ≤ FullScan = истинный
  max`. У `ker_atmos` максимум на границе блока (у диагонали), т.е. текущий `rawmax` уже =
  истинному максимуму. Сделать его МЕНЬШЕ истинного max нельзя — это нарушит мажоранту
  (`accept>1` → смещение). Accept-rate 25% фундаментален для ядра.
- **N8b (меньший `min_block` под N7) — ❌ mb=8 всё равно оптимум.** Гипотеза: N7 удешевил
  D2.3, значит меньший mb (меньше событий) теперь выгоден. Не подтвердилась: штраф за малый
  mb сидел в **D2.2** (больше блоков/accept), а его N7 не трогает. Свип 2/4/8/16 →
  124/112/108/114 с, минимум на mb=8.

Вывод: одно-поточная микро-оптимизация выжата. Остаток wall — латентность рассеянного
(D2.2) и зависимого (C2) доступа, по которым рычаги испробованы. Дальше — только
параллелизм или смена алгоритма (см. Часть VI).

### Вставка — два эпизода про инженерную аккуратность

- **Тестовый бенчмарк (5 сидов, mean±std).** Финальный замер: fast-бинарь,
  последовательно (чистый wall), 5 сидов, t=1e3…1e7. CV (std/mean) = **0.1–1.2%** —
  тайминг крайне воспроизводим. lazy 0.05 даёт −18…−27% (пик −27% на 1e5, спад до −18% на
  краях: на 1e7 система огромна и D2.3 относительно меньше).
- **Свип по объёму `V` и баг переполнения.** Добавили параметр `V_INIT`. При V=10⁷→10⁸
  Frobenius падает в **√10 ≈ 3.16×** на всех t (0.0146→0.0045, 0.0311→0.0100,
  0.0678→0.0213, 0.1517→0.0481) — это закон **Frob ∝ 1/√V** (учебниковая MC-сходимость:
  шум ∝ 1/√N_частиц; заодно независимое подтверждение несмещённости N7 при V=1e8). При
  этом всплыл **баг**: счётчики `total_events`/`accepted` были `int` и при V≥10⁸ (события
  >2.1·10⁹) переполнялись (отрицательное число!). Исправлено на `int64_t`. **Важно: это
  только отчётный счётчик — цикл идёт по ВРЕМЕНИ, физика/Frobenius были верны и при
  переполнении** (Frobenius совпал до и после фикса). Прежние V=10⁷ результаты не затронуты.

## Часть IV. Сквозные уроки (мета-выводы)

1. **Профилируйте, не угадывайте — и на ЦЕЛЕВОМ железе.** opt-7 дал 1% на одной машине и
   19.5% на другой. Несколько «очевидных» идей провалились (opt-10/11/14), а недооценённые
   (opt-7, N6) выстрелили. Интуиция систематически ошибалась в разы.
2. **Поймите ПРИРОДУ узкого места до оптимизации.** Мы latency-bound на зависимых цепочках
   (§2.2). Это объясняет И победы (B-арность, MLP), И провалы (float N2, SoA opt-11,
   prefetch opt-14) одним принципом.
3. **«Алгоритмически лучше» ≠ «быстрее».** opt-5 (sort) асимптотически лучше, но медленнее
   на малых `K`. Константы и предсказуемость ветвлений решают.
4. **Смотрите ассемблер/codegen.** opt-10 (cbrt) был избыточен — компилятор уже это делал.
5. **Разным под-задачам — разные структуры.** B-арность помогла дереву интенсивностей и
   навредила дереву частиц (N3 ✅ vs N4 ❌) — из-за разного распределения глубины спуска.
6. **Деградация к тривиальному параметру — мощный тест.** «B=2 должно совпасть с бинарным
   бит-в-бит» поймало 2 бага в opt-18.
7. **Дисциплина отката с уроком.** Каждый провал задокументирован с причиной — это ценнее
   будущему, чем тихо оставленный «+0.3% шум».
8. **Меняете критерий корректности осознанно.** Переход от «бит-в-бит» к «статистически
   несмещён» (N7) открыл крупнейший выигрыш — но потребовал доказательства + 50-сидовой
   проверки + опционального флага, чтобы не терять проверяемый baseline.

## Часть VI. Где мы сейчас и куда дальше (roadmap)

### Потолок одного потока

Мы в **~10–20% от физического пола** этого алгоритма на этом CPU. stall ≈ 23% тактов —
это чистое ожидание памяти на зависимых цепочках спуска/обновления; полностью убрать его
нельзя (латентность зависимых обращений неустранима). Реалистичный практический пол
известными безопасными методами — около текущих 107 с. Остаток — memory-latency по D2.2
(рассеянный доступ к блокам) и C2 (зависимый спуск), по которым рычаги испробованы и
упёрты. Микро-оптимизация одного потока **исчерпана**.

### Направление 1 — параллелизм «1 симуляция на ядро» (максимальный ROI, низкий риск)

Симуляции с разными сидами независимы. Запуск по одной на ядро даёт ~×32 throughput на
32-ядерной ноде. Option D (opt-19) это уже подготовил: горячий цикл строго одно-поточный,
лишних потоков нет. 50-сидовая проверка уже гонялась так (24 параллельно) — работает из
коробки. Нужна лишь обвязка (запуск N сидов на ядра + сбор статистики). Это и есть
естественный следующий шаг.

### Направление 2 — GPU (нюансировано: не серебряная пуля)

Ядро одного прогона — **анти-паттерн для GPU**: последовательная зависимость событий
(нераспараллеливаема внутри прогона), дивергентный pointer-chasing (расхождение варпов,
некоалесцированный доступ), а латентность HBM ВЫШЕ, чем у L2/L3 CPU. Портирование 1:1 будет
медленнее per-sim. Где GPU реально может выиграть:

- **Ансамбль независимых симуляций** (тысячи сидов, латентность прячется между ними) — но
  только на **малых/средних t**, где рабочий набор мал и тысячи прогонов влезают в память
  GPU. На больших t (ГБ/прогон) влезает <20 прогонов → латентность не спрятать → CPU
  выигрывает. Потенциал на малых t: ×5–20 throughput, но недели-месяцы CUDA-порта.
- **τ-leaping** (см. ниже) — вскрывает data-parallelism, тогда GPU заиграет и на одном
  прогоне.

### Направление 3 — τ-leaping (крупный, но приближённый)

За один шаг `τ` разыгрывать сразу ПАЧКУ коагуляций (число каждого канала ~Пуассон) вместо
одного события. Вскрывает параллелизм (по блокам/каналам), потенциально большое ускорение —
но это **приближённый** метод (смещение `O(τ²)`, нужен контроль ошибки и разрешение
конфликтов одновременных обновлений `N`). Большой рерайт. Раз статистический режим уже
принят (N7), философски на столе. Выгоден и на CPU, и на GPU.

### Сводка приоритетов

1. **Параллелизм 1-сим/ядро** — дёшево, надёжно, ×десятки. Делать первым.
2. **GPU-ансамбль малых t** — если нужны массовые статистические/параметрические прогоны.
3. **τ-leaping** — самый высокий потолок, но приближённо и дорого по разработке.

## Приложение A. Глоссарий

- **Wall (wall-clock time)** — реальное затраченное время прогона.
- **Frobenius error** — относительная норма отклонения распределения от эталона; метрика
  корректности.
- **Accept-Reject (выборка с отказами)** — выбор из сложного распределения через выборку из
  простого мажорирующего + случайное принятие; отказы корректируют завышение.
- **Мажоранта** — верхняя оценка (`rawmax_b ≥ K`, `Î_b ≥ I_b`).
- **Прореживание Пуассона (thinning, Ogata)** — генерация точечного процесса с переменной
  интенсивностью через мажорантный процесс + принятие; основа корректности AR и N7.
- **Сегментное дерево** — структура для «выбрать ∝ весу» и обновить вес за `O(log)`.
- **Ленивое масштабирование (lazy scale)** — умножить все веса на константу за `O(1)`.
- **B-арное дерево** — у каждого узла B детей; глубина `log_B N` (короче бинарного при B>2).
- **MLP (memory-level parallelism)** — несколько независимых промахов памяти «в полёте»
  одновременно; перекрывает латентность.
- **Latency-bound / bandwidth-bound** — упёртость в задержку отдельного обращения к памяти
  vs в суммарную пропускную способность. Мы — latency-bound.
- **pointer-chasing** — последовательность зависимых разыменований (спуск по дереву): каждый
  шаг ждёт предыдущего; худший случай для латентности.
- **AoS / SoA** — Array-of-Structs / Struct-of-Arrays; раскладка данных в памяти.
- **CSR (Compressed Sparse Row)** — плоское представление списков (offsets[] + data[]) без
  отдельных аллокаций на строку.
- **IPC** — инструкций за такт (мера утилизации конвейера).
- **stall** — такты, в которых процессор не выполнил ни одной микрооперации (ждёт).
- **PGO (profile-guided optimization)** — компиляция с учётом профиля частот ветвлений.
- **`-march=native` / `-ffast-math`** — флаги: код под конкретный CPU (AVX-512 и т.п.) /
  более агрессивная (но менее «точная») плавающая арифметика.
- **rdtsc** — инструкция чтения счётчика тактов CPU (для ручного профилирования фаз).
- **min_block** — минимальный размер блока в Mosaic-разложении (компромисс tightness ↔
  число блоков).
- **Population rescaling** — удвоение системы при падении числа частиц вдвое (контроль
  статистики).
- **HBM** — высокоскоростная память GPU (большая полоса, но высокая латентность).
