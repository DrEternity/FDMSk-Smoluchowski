# ANALYSIS.md — журнал оптимизации FDMSk-Smoluchowski

> Журнал гипотез, наблюдений и экспериментов по ускорению **детерминистического**
> решателя уравнения Смолуховского методом конечных разностей (FDM) с
> mosaic-skeleton (MSk) сжатием матрицы ядра. Аналог документа
> `OPTIMIZATION_STORY_MONTECARLO.md`, но задача здесь другая (не Монте-Карло, а
> интегрирование ОДУ), поэтому и узкие места другие — их ищем профилированием и
> прогонами в slurm.
>
> **Дисциплина (как в MC-проекте):** ветка `optimization-sweep`, по коммиту на
> успешный шаг. Оптимизация принимается, только если (1) wall упал заметно
> (>~2%, выше шума ноды); (2) решение совпадает с эталоном `new_reference_*` в
> пределах ODE-допуска (Frobenius-отклонение ≲ 1e-6, см. §Метрики). Провалы
> документируем с уроком.

## Часть I. Что считаем (постановка)

Уравнение коагуляции Смолуховского решается как система ОДУ на концентрации
`n_k(t)` (моно-дисперсное начальное условие: `n[0]=1`, остальные 0). Правая
часть (RHS) для каждого `k`:

```
dn_k/dt = 1/2 · Σ_{i+j=k} K(i,j) n_i n_j   −   n_k · Σ_j K(k,j) n_j
          \_______ gain (коагуляция в k) ___/      \__ loss (уход из k) __/
```

- **gain-член** = дискретная свёртка `(K∘n) * n` — в коде `_matrix.convolve(...)`.
- **loss-член** = матвек `n_k · (K·n)` — в коде `_matrix.matvec(...)`.

Матрица ядра `K` размера `N×N` плотная, но гладкая вдали от диагонали → её
сжимают **mosaic-skeleton** разложением (библиотека MSk, та же `zaimsk`, что и в
MC-проекте). Свёртка по блокам делается через FFT (Тёплицевы блоки),
матвек — через BLAS (low-rank/dense блоки).

Интегратор — **адаптивный RK4** (`runge_kutta4_adaptive_mass` в
[python_var.hpp](python_var.hpp)) с контролем шага по локальной ошибке (tol =
`ode_tol`) и контролем «фронта»: когда самый крупный заметный размер частицы
доходит до `size/2`, матрицу **удваивают** (resize), пересобирая MSk-аппроксимацию
для нового размера. Старт `initial_size=512`, потолок `max_size=1048576`.

### Ядро (атмосферное)
`kernel_atmos(i,j)` (1-индексное), [main.cpp:12](main.cpp#L12):
```
K(i,j) = (i+j)·(i^{1/3}+j^{1/3})^{2/3}  /  ( (i·j)^{5/9} · |i^{2/3} − j^{2/3}| ),   i≠j
K(i,i) = 4
```
**6 вызовов `std::pow` на одну оценку** (как в MC-ядре до мемоизации, ср. MC N6).

## Часть II. Как получены референсные решения (разобрано)

Решения лежат в `build/` (по 24 МБ каждое, 1048576 строк). Две серии:

| Префикс | ODE tol | Назначение |
|---|---|---|
| `reference_solution_atmos_T*.txt` | **1e-6** | быстрый прогон |
| `new_reference_solution_atmos_T*.txt` | **1e-9** | точный эталон («true» reference) |

**Единственное отличие** серий — допуск RK4 (`ode_tol`): у `new_*` он жёстче
(1e-9 против 1e-6) ⇒ мельче шаг ⇒ ~4× дольше, но точнее. Это и есть «эталонный
эталон» для проверки точности.

**Команда генерации** (восстановлено из `build/slurm-18543/18544.out` +
`build/cluster_launch.sh`):
```bash
MAX_SIZE=1048576
for T in 10 100 1000 10000 100000 1000000; do
    ./example ${MAX_SIZE} ${T} new_reference_solution_atmos_T${T}.txt 512 0.01 1e-9
done
# аргументы example: max_size  time  output  initial_size  mass_threshold  ode_tol
```
Конфигурация решателя при этом (working-tree main.cpp/python_var.hpp, НЕ
закоммиченный HEAD — он устарел): `kernel_atmos`, `MosaicType::tridiag` (rho=2.0),
`min_rows=min_cols=128`, MSk `rel_tol=1e-10`. Slurm: partition `c32m384`,
`--cpus-per-task=16`.

## Часть III. Профиль стоимости (главное!)

Разбор T=10⁶ прогона `reference` (slurm-18543) по фазам — суммарно по эпохам
размера:

| size | approx (построение MSk), c | RHS evals × avg = Σ RHS, c |
|---:|---:|---:|
| 512 | 0.10 | 6132 × 0.0008 = 4.97 |
| 1024 | 0.25 | 180 × 0.0024 = 0.44 |
| 2048 | 0.57 | 116 × 0.0069 = 0.80 |
| 4096 | 1.25 | 72 × 0.020 = 1.43 |
| 8192 | 2.70 | 44 × 0.047 = 2.06 |
| 16384 | 5.72 | 32 × 0.226 = 7.23 |
| 32768 | 11.9 | 24 × 0.525 = 12.6 |
| 65536 | 25.1 | 16 × 1.263 = 20.2 |
| 131072 | 52.7 | 16 × 3.277 = 52.4 |
| 262144 | 110 | 8 × 8.855 = 70.8 |
| 524288 | 236 | 12 × 25.70 = 308 |
| 1048576 | 495 | 4 × 103.1 = 413 |
| **Σ** | **~941 (51%)** | **~894 (49%)** |

**Вывод 1.** Время делится **≈ пополам** между построением MSk-аппроксимации и
интегрированием (RHS). Оптимизировать надо ОБЕ фазы.

**Вывод 2.** Стоимость резко растёт с размером: последние две эпохи (524288 +
1048576) = ~1440 c из 1835 c (**78%**). Для цели t=10⁷ (где size дойдёт до
~4–8 млн) именно крупные эпохи будут доминировать.

**Вывод 3 (внутри RHS).** `convolve` (gain) ≫ `matvec` (loss): на size=16384
конв ~157 мс против матвек ~8.6 мс (**~18×**). Свёртка — главный потребитель
внутри RHS. (Числа с login-ноды, шумные, но порядок ясен.)

### Рост фронта и выбор `max_size` для t=1e7 (КЛЮЧЕВОЕ для массы)

Из t=1e6-прогона (slurm-18544) — моменты `t`, когда фронт достигает `size/2` и
триггерит resize `S→2S`:

| size S | фронт→S/2 при t≈ | отношение к пред. |
|---:|---:|---:|
| 2^19=524288 | 4.55e5 (замер) | — |
| 2^20=1048576 | 1.2e6 (экстрап.) | ×2.7 |
| 2^21 | 3.3e6 | ×2.7 |
| 2^22 | 9.0e6 | ×2.7 |
| 2^23=8388608 | **2.4e7** | ×2.7 |

Время триггера растёт **×~2.7 на удвоение** (характерный size ∝ t^0.70). При t=1e6
финальный size был 1048576, масса = **0.999999** (сохранена — фронт не дошёл до
границы). Для **t=1e7**: фронт дойдёт до ~2.25M, последний resize 2^22→2^23 при
t≈9e6, дальше до t=1e7 size=2^23 (след. триггер 2^24 только при t≈2.4e7).

⇒ **Для t=1e7 нужен `max_size = 2^23 = 8388608`.** Тогда фронт < size/2=4.19M, ни
одна частица не «утекает» за границу сетки ⇒ масса сохраняется. С текущим
`max_size=2^20` прогон упрётся в границу ~t=3e6 и начнёт терять массу. Память при
2^23: ~10 векторов × 67 МБ ≈ 0.7 ГБ + MSk-структура — влезает в 376 ГБ ноды.

## Часть IV. Гипотезы оптимизации

### H1 (ГЛАВНАЯ) — весь прогон идёт в ОДИН поток. `n_jobs=1`.
В [main.cpp:93,247](main.cpp) (и в python_var.hpp:247) MSk строится вызовом
`MosaicSkeleton<double>::approximate(matrix, oracle, 1)`. Третий аргумент —
`n_jobs` (дефолт **4**), это размер пула потоков `p_pool`.

Проверено по исходникам MSk
([mosaic_skeleton.hpp](external/mosaic-skeleton/include/msk/mosaic_skeleton.hpp)):
- `approximate(..., n_jobs, ...)` создаёт `ThreadPool` из `n_jobs` воркеров.
- Построение блоков (`build_mosaic`/`approximate`) использует этот пул.
- **И `matvec`, и `convolve`** диспатчат блоки через
  `apply_batched_block_linear → p_pool->execute(...)` — тот же пул.

⇒ при `n_jobs=1` **и аппроксимация, и весь RHS — однопоточные**, при том что
slurm выделял 16 ядер (а на ноде их 32 физических). Это ожидаемо крупнейший
выигрыш: поднять `n_jobs` до числа ядер. Ограничение — закон Амдала (часть RK4
последовательна: clamp-цикл, combine-цикл, `BLAS::axpy/copy/nrm2` по O(size)).
**Статус: ✅ ПОДТВЕРЖДЕНО (sweep 2026-06-07, см. §Журнал).** На T=10000 (size→65536)
масштабирование `rel_nj1→nj16` = **3.4×**, насыщение при nj≈8–16, регресс при nj=32
(оверсабскрайб на скромном размере). Амдаловская доля serial ≈19% (8.2 c из 43.8 c)
— это BLAS-1 векторные операции RK4. На больших size (миллионы, цель t=1e7) parallel-
доля растёт (convolve/approx ∝ size·polylog ≫ serial ∝ size) ⇒ nj=16–32 даст больше.
**Масса при всех nj идентична (0.99999902), Frobenius между nj ≲ 5e-13** — параллелизм
физику не меняет.

### H2 — сборка идёт на `-O0` (нет флагов оптимизации).
`CMAKE_BUILD_TYPE` пуст ⇒ `example`, `bench` и ядро `zaimsk` компилируются с
`CXX_FLAGS = -std=gnu++11 -fopenmp`, **без `-O3/-march`**. (Только под-библиотеки
`cpp_blas`/`cpp_lapack` имеют свой захардкоженный `-O2`.)
- Горячие `convolve`/`matvec` — это header-only шаблоны, инстанцируются в нашем
  TU ⇒ компилируются на `-O0`.
- **НО** первый микро-бенч (login-нода, шумно) показал: `-O3 -march=native`
  почти НЕ ускоряет matvec/convolve (они упёрты в FFT/BLAS из готовых `.so`).
  approx-фаза от `-O3` улучшилась на ~10–35% (шумно). Урок (как в MC): профиль
  важнее интуиции — «-O3 даст кратный выигрыш» неверно для FFT/BLAS-bound кода.
- `-march=native` безопасен (сборка = запуск, тот же Cascade Lake/AVX-512).
- `-ffast-math` рискован для ЭТАЛОНА (меняет FP-результат) — отложить/проверять
  отдельно на точность.
**Статус: ✅ ПОДТВЕРЖДЕНО (sweep 2026-06-07).** Чистый замер на slurm-ноде:
`Release -O3 -march=native` vs пустой build type (-O0), оба 1 поток, T=10000 →
**1.72×** (75.4 c → 43.8 c). Масса идентична (0.99999902), Frobenius vs -O0 = 4.6e-13.
Закоммичено в `CMakeLists.txt` (дефолт Release + опция `SMOL_NATIVE=ON`).
`-ffast-math` НЕ включён (бережём эталонную точность). На крупных size выигрыш от
march может быть меньше (FFT/BLAS из готовых `.so`), но на approx-фазе он реален.

### H3 — `convolve` (FFT) доминирует внутри RHS.
Если так, рычаги: (а) параллелизм по блокам (H1 покрывает); (б) FFTW с
несколькими потоками; (в) число/структура Тёплицевых блоков (min_block, rho).
**Статус: гипотеза, мерить долю convolve в RHS отдельно.**

### H4 — ядро передаётся как `std::function` ⇒ нет инлайна.
`modeling(..., std::function<double(double,double)> kernel, ...)` оборачивается в
лямбду `new_kernel`. На фазе аппроксимации ядро сэмплируется МНОГО раз; каждый
вызов = диспетч `std::function` + 6 `std::pow`. Шаблонизация ядра (как MC opt-0)
может ускорить построение. Трогаем только наш код, не MSk. **Статус: гипотеза.**

### H5 — мемоизация ядра (6 `std::pow` → таблицы).
4 из 6 pow — от целого размера (`i^{1/3}, j^{1/3}, i^{2/3}, j^{2/3}` и т.п.).
Кэш `pow`-таблиц (как MC N6, −6.6%) ускорил бы построение MSk. Трогаем только
наш kernel-код. **Статус: гипотеза, после H1/H2.**

### H6 — FFTW multithreading.
FFTW может считать один план в несколько потоков (`fftw_plan_with_nthreads`).
Если convolve упёрт в один большой FFT на крупный блок, это могло бы помочь
независимо от блочного параллелизма. Зависит от того, линкуется ли `libfftw3_threads`.
**Статус: гипотеза, низкий приоритет (H1 уже даёт блочный параллелизм).**

### H7 — алгоритм/точность интегратора.
`new_reference` (ode_tol 1e-9) делает в ~4× больше шагов. Для самого эталона это
оправдано, но если нужны менее точные прогоны — ode_tol 1e-6 уже в 4× быстрее.
Не «оптимизация кода», но рычаг для t=10⁷. **Статус: вне scope кода.**

### H8 (КРИТИЧНО для t=1e7) — ПАМЯТЬ упирается в потолок ноды; рычаг = `rel_tol`.
**Открыто 2026-06-07 при первой попытке t=1e7 (job 44950): OOM (SIGKILL) при входе
в size 2097152 (2^21).** Замер пиковой памяти (mem_probe) по size при rel_tol=1e-10:

| size | peak RAM | × к пред. |
|---:|---:|---:|
| 262144 | 17.9 ГБ | — |
| 524288 | 47 ГБ | 2.6 |
| 1048576 | **158 ГБ** | 3.4 |
| 2097152 | ~470 ГБ (экстрап.) | → OOM (нода 376 ГБ) |

Пик — это **транзиент фазы построения** `approximate()` (память скачет 22→158 ГБ
на build, потом оседает). Память растёт **×~3 на удвоение size** (быстрее 2× — рост
суммарного ранга блоков). ⇒ 2^22 ≈ 1.4 ТБ, 2^23 ≈ 4 ТБ — **на одной ноде 376 ГБ
size 2^21+ недостижим при rel_tol=1e-10**.

**Пересмотр цели по массе:** к t=1e7 фронт доходит лишь до ~2–2.5M (Часть III'),
поэтому **`max_size=2^22=4194304` достаточно** (верх сетки 4.19M ≫ фронт) — 2^23 НЕ
нужен. Но 2^22 при 1e-10 = 1.4 ТБ ⇒ нужно ~4× меньше памяти.

**Рычаг — `rel_tol`** (точность MSk-аппроксимации, env `MSK_REL_TOL`, добавлен в
main.cpp). Механизм найден в API: `daemon.init_memory(n_threads+1, max_size, max_rank)`
— per-thread скретч фазы build ∝ РАНГУ блоков. При 1e-10 ранги огромны ⇒ скретч
огромен; при 1e-5 ранги малы ⇒ скретч крошечный. Скретч освобождается после build
(потому 158→22 ГБ).

**✅ ПОДТВЕРЖДЕНО (probes peak_mem + accuracy, 2026-06-07):**
- Пик памяти при rel_tol=1e-5 (direct build, nj16): **size 2^20 = 18 ГБ, 2^21 = 29 ГБ**
  (против ~470 ГБ при 1e-10 → тот самый OOM). **n_jobs на память НЕ влияет** (18/17/17
  ГБ при nj=16/4/1) ⇒ держим nj=16 ради скорости.
- **Точность не страдает:** T=1000, Frobenius(1e-5 vs 1e-10) = **1.1e-7**, vs эталон
  T1000 (ode 1e-9) = 2.4e-6. **Масса идентична 0.9999995** при всех tol. Даже быстрее
  (T=1000: 4.06→3.21 c).
- **Почему mem_lever на 524288 не увидел эффекта:** там пик памяти — это integration
  (~47 ГБ, tol-независимо), а build (tol-зависимый) был НИЖЕ. На крупном size build
  доминирует ⇒ tol решает.

⇒ **t=1e7 теперь по памяти реален при nj=16: rel_tol=1e-5, max_size=2^23 (~≤120 ГБ).**

### H8' — УТОЧНЕНИЕ (2026-06-07, по живому прогону 44962): два разных источника памяти
Первичный диагноз «OOM = build scratch» оказался НЕПОЛНЫМ. На полном прогоне (resize-
путь) при rel_tol=1e-5 память на size 1048576 всё равно вспухает до **154 ГБ** — это
НЕ build (build там 18 ГБ, подтверждено direct-probe), а **транзиент ФАЗЫ ИНТЕГРАЦИИ**
(`convolve`). По коду `convolve` (mosaic_skeleton.hpp ~1266): каждый блок пишет в свой
`z_local` через пул потоков ⇒ суммарно ~ (число блоков × O(size)) — отсюда десятки-сотни
ГБ. **Эта память освобождается на каждом resize** (видно: 1048576=154 ГБ → resize →
2097152=24 ГБ). Т.е. пик — per-epoch транзиент, не накопление.

**Два независимых источника памяти, оба per-epoch:**
1. **Build scratch** ∝ ранг ⇒ управляется `rel_tol` (1e-10: огромный; 1e-5: крошечный).
2. **Integration `convolve` workspace** ∝ (число блоков × size), от tol почти НЕ зависит.

**Почему attempt#1 (1e-10) умер на 2^21, а 44962 (1e-5) — нет:** на 2^21 при 1e-10
build-scratch (сотни ГБ) + integration-spike вместе > 376 ГБ → OOM. При 1e-5 build-
scratch ≈ 0 ⇒ остаётся только integration-spike, и на входе в 2^21 он всего **31.7 ГБ**
(flat) ⇒ прошёл. Т.е. rel_tol=1e-5 НЕ убирает integration-память, но освобождает
бюджет, которого хватило пройти узкое место.

**Остаточный риск ПОДТВЕРДИЛСЯ → OOM:** прогон 44962 (rel_tol=1e-5) тоже умер на 2^21
(`rc=137`=SIGKILL), но позже attempt#1 — на ШАГЕ интегрирования 2^21 (память была flat
31.7 ГБ, затем convolve одного шага вспух >376 ГБ между сэмплами). Т.е. **integration
convolve workspace — фундаментальная стена**: 1048576 = 154 ГБ, 2^21 > 376 ГБ,
tol-НЕзависимо. rel_tol=1e-5 лишь отсрочил смерть (убрал build-компоненту).

**Важное наблюдение про вход в эпоху:** на ВХОДЕ в 2^21 (фронт ~847k, узкий) память
была 31.7 ГБ — convolve-спайк зависит от ШИРИНЫ ФРОНТА (активной зоны), а не от size:
direct-probe 1048576 с IC n[0]=1 (разреженный вход) = 18 ГБ, а полный прогон с широким
фронтом = 154 ГБ. Пик эпохи наступает, когда фронт расширяется к size/2 перед resize.

**⇒ Ключевой нерешённый вопрос: convolve workspace — per-thread (n_jobs) или per-block?**
Если per-thread — снижение n_jobs спасает (медленно, но влезет). Если per-block — n_jobs
не поможет, и t=1e7 на одной ноде этим методом нереален (нужен другой convolve / MPI —
вне scope). **Статус: ПРОВЕРЯЮ n_jobs-зависимость integration-памяти (sweep на 1048576).**
Первый замер (job 44965): nj=16 → пик **150.3 ГБ** (подтверждает 154). Ждём nj=4/1.

### H9 (НОВЫЙ рычаг памяти) — min_block. ДО СИХ ПОР НЕ ВАРЬИРОВАЛСЯ.
Текущий `min_block=128` (min_rows=min_cols), жёстко в python_var.hpp — все мои свипы его
НЕ трогали (env-ручки были только MSK_NJOBS, MSK_REL_TOL). bench.cpp перебирает {32..512},
но это отдельный микробенч, не решатель.

**Почему это прямой рычаг на стену памяти:** пик ≈ `8B · Σ_блокам (N−(i0+j0))`, а число
блоков ∝ 1/min_block. При слиянии 4 мелких блоков в 1 крупный остаётся ОДИН буфер длины
`N−(i0+j0)_min` вместо четырёх ⇒ Σ падает кратно. Т.е. **больше min_block → кратно меньше
буферов свёртки → ниже пик.** Цена: крупнее плотные диагональные блоки (хуже сжатие, чуть
больше счёта у диагонали; но dense-блоки ТОЧНЫ ⇒ точность не падает, может вырасти).

Добавлена env-ручка `MSK_MIN_BLOCK` (python_var.hpp, как MSK_NJOBS).

### H8'/H9 — РЕЗУЛЬТАТЫ ПРОФИЛИРОВАНИЯ (2026-06-07/08). КОРЕНЬ НАЙДЕН.
Профилирование: (а) свипы памяти на slurm; (б) разбор кода `apply_batched_block_linear`
+ `MemoryBank`; (в) свой malloc-интерпозер (LD_PRELOAD, sweeps/memtrace.c).

**Корень:** в `convolve` каждый блок пишет в СВОЙ выходной буфер длины `N−(i0+j0)`
(`MemoryBank::get` → malloc/realloc, кэшируется до конца свёртки). Все батчи блоков
ставятся в пул СРАЗУ, а сворачивает их ОДИН поток-аккумулятор ⇒ буферы копятся «в
полёте». Пик ≈ (#блоков) × (доля-pile-up от n_jobs). Интерпозер подтвердил: пик — это
крупные (мега-байтные) буферы (бакеты 2^21–2^24) + ~8 ГБ FFTW через mmap, НЕ мелочь.

**Два независимых рычага (замеры на size 1048576, rel_tol=1e-5, масса везде 0.99765):**

| рычаг | значения → пик памяти |
|---|---|
| **n_jobs** (pile-up) | 16 → 150 ГБ; 4 → 111 ГБ; **1 → 18 ГБ** |
| **min_block** (#блоков) | 128 → 150 ГБ; 256 → 86; 512 → 61; **1024 → 51 ГБ** |

- **n_jobs** нелинеен: вся беда от pile-up; при nj=1 его нет → 18 ГБ (×8.4 меньше).
- **min_block** ×8 (128→1024) даёт ×3 память И ×2 СКОРОСТЬ (471→232 c), **точность не
  страдает** (Frobenius vs mb=128 = 3.6e-7, масса идентична). Чистый выигрыш.
- Рычаги **комбинируются** (min_block ↓#блоков, n_jobs ↓pile-up) ⇒ для 2^22 (нужно
  <376 ГБ) масса вариантов: напр. mb=1024+nj=1 ≈ 6 ГБ@2^20 → ~60 ГБ@2^22 (с огромным
  запасом), или mb=1024+nj=2/4 быстрее с приемлемым запасом.

⇒ **t=1e7 РЕШАЕМ на одной ноде.** Калибрую nj-фактор при mb=1024 и выбираю самый быстрый
combo, гарантированно влезающий в 2^22, затем финальный запуск. Закоммичено opt-3
(MSK_MIN_BLOCK). **Это НЕ правка библиотеки MSk — только env-параметры нашего кода.**

   одновременного спроса и не ужимается ⇒ RSS «защёлкивается». Ограничено (плато ≈
   пик-воркспейс × фрагментация), не утечка. Пик к концу был 152 ГБ (а не 250–320, как
   пессимистично оценивал: у 2^23 фронт узкий относительно size ⇒ воркспейс умеренный).
   **Митигация (если упрётся): env `MALLOC_MMAP_THRESHOLD_=131072 MALLOC_TRIM_THRESHOLD_=131072`**
   → крупные буферы всегда mmap и возвращаются ОС при free ⇒ RSS осциллирует, не растёт.

**Масса 0.99977 (дрейф 2.3e-4):** копится за счёт несимметричности `K̂` (аппроксимация)
+ клампинг отрицательных за 1e7 времени. На эталонах t≤1e6 было ~1e-6; рост дрейфа с t
ожидаем. При желании уменьшить — тоньше ode_tol/rel_tol (дороже). 2.3e-4 = «не ушла далеко».
